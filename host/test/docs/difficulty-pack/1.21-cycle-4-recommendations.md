## Cycle 5 — needle-haystack manifest-schema bug

Sweep ran 6 cells; all 6 reps PASSED in 8-11s wallclock; all 6 registry-emit attempts FAILED with:

```
TestManifestValidationError: test manifest in /test/__tests__/tier-eval/needle-haystack.test.js
failed validation: field secondary_axes: array value "retrieval_over_distance" not in enum
```

**Root cause:** `host/test/lib/schemas/test_manifest.schema.json` defines a closed enum for `secondary_axes` items (`spec_precision, stateful_logic, multi_file_context, convergence, tool_discipline, productivity, local_usability`). I authored needle-haystack with `retrieval_over_distance` (a descriptive label that doesn't exist in the schema).

**Fix:** changed `secondary_axes` to `["tool_discipline"]`, which is the closest in-schema match (the test does exercise tool selection: grep vs read-files-individually). Manifest now validates; cycle 6 produced 6 valid registry rows.

**Sharp edge for future test authoring:** the schema isn't validated at author-time. The author can write any string and only discover the validation failure when the registry-emit path runs (which only happens during a sweep with `RUN_REGISTRY_EMIT=1`). A pre-commit smoke test that calls `validateManifest` on every test file under `__tests__/tier-eval/` would catch this in seconds. Logging as a 1.22 follow-up.

## Cycle 6 — needle-haystack v1 saturates trivially

### Per-cell

| test_id | t16 | t32 | p90 iters | p90 elapsed |
|---|---|---|---|---|
| `needle-haystack` | 100% (3/3) | 100% (3/3) | 8 (t16) / 6 (t32) | 12.8s / 12.6s |

Both tiers ceiling-pinned. R1 (saturation) + R6 (>85% pass) flagged for both. **Not a discriminator at this difficulty.**

### Why v1 doesn't discriminate

The plan predicted t16 30-70% / t32 80-100% under a model that "takes a less efficient retrieval strategy" at lower quant. C6 evidence shows the model uses `grep -rn REGION_KEY lib/ data/ config/` in iteration 1 and writes solve.js in iteration 2. tool_discipline is fine at IQ4_XS for this task; ctx never gets pressured because grep doesn't load file bodies.

The 30-files × 1.25kb haystack is a **find-a-string-with-grep** test, not a **multi_file_context** test. It saturates universally because grep is universal.

### Options for v2 (need user decision)

The base discrimination signal we want: t16 should be materially worse than t32 at this task. To get there, the task has to either (a) defeat single-grep, or (b) require enough sustained reasoning that IQ4_XS slips. Six options, ordered by how much they preserve the original v1 design:

1. **(Lightweight) `grep` returns ambiguous matches.** Distractor names that contain `REGION_KEY` as a substring (`REGION_KEY_BASE`, `REGION_KEY_HASH`, `REGION_KEY_OFFSET`) — coarse `grep REGION_KEY` returns 30 hits, model has to read each match line to identify the un-prefixed/un-suffixed real export. Probably still trivially solved by `grep -wn` or by reading line content from grep output. Low confidence in discrimination.

2. **(Medium) Split-needle.** `REGION_KEY = REGION_KEY_PART_A + REGION_KEY_PART_B + REGION_KEY_PART_C` across three files; model has to find all three parts AND get the concatenation order right. Single grep returns 3 hits, model has to parse + concat correctly. Still solvable in ~3 iter; modest difficulty bump.

3. **(Medium-strong) Indirect lookup.** Needle file contains `export const REGION_KEY = LOOKUP[REGION_KEY_INDEX];` where `REGION_KEY_INDEX = 7` lives in another file and `LOOKUP = [...]` lives in a third. Model has to chase the chain (grep doesn't reveal the value, only the formula). 4-5 iter required. Better discriminator but still grep-able if model reads the matching lines.

4. **(Strong) Bury the needle in a function.** Replace `export const REGION_KEY = '...'` with an exported function that COMPUTES the value from a hash of fields scattered across multiple files. Model has to read enough of the source to understand the computation, then write solve.js that REPLICATES it. This shifts the test from retrieval to comprehension — closer to a real multi_file_context probe.

5. **(Reframe) Drop needle-haystack v2; author a different multi_file_context probe.** Maybe a directory-tree-restructure task: rearrange a flat workspace into a specified package structure (move + update imports). Genuinely tests multi-file reasoning. But this is essentially a new test, not a v2.

6. **(Backstop) Bump file count to 60-100 files.** Raises the haystack to 18-30k tokens; if the model insists on grep this still saturates. But if any reps fall back to `read_file` calls (e.g., if grep is unavailable or returns errors), ctx pressure is real. Cheapest variant; least confident in discrimination.

**Recommendation: option 3 (indirect lookup) for v2** — preserves the NIAH framing, requires multi-step retrieval, no infrastructure changes. Hand-solve still ~5 min. If c7 pilot saturates again, escalate to option 4.
