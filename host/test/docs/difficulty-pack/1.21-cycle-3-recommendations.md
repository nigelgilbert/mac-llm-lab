## Forensics — the three Q's that drove the c3 read

### Q1 — Cascade-eight 9-file bump did NOT fire

**Mechanism:** Model edited `run.js` to remove `import { contains } from './i.js'` despite the prompt's "Do not edit run.js" instruction. There was no enforcement. Workspace listing at end of run shows only `a.js…h.js + run.js`; no `i.js`. The Explore agent confirmed 0 `read_file('/workspace/i.js')` calls across all 6 reps.

**Implication:** All c3 cascade-eight data reflects the *pre-bump* 8-file test, not the intended NaN/indexOf trap. The 100% saturation reading is correct for the 8-file version, but says nothing about the 9-file version.

**Fix:** Add a `run.js` SHA256 checksum gate in `beforeEach` → compare against on-disk hash post-claw → fail loudly with descriptive message if mismatch. Iterating in cycle 4. ~10 LOC.

### Q2 — Book-store t32 67% is bridge telemetry loss, not real failure

**Mechanism:** Run `eca7672b-2c59-4c68-99cc-545ce0453c28` shows `terminal_status: timeout`, `join_status: stream_aborted_mid_run+count_mismatch`, `run_elapsed_ms: 719110` (12 min wallclock vs 285s nominal timeout), all per-iter `request_*_ms` fields null. Same shape as 2 other c3 stalls (ini-parser t32 rep2 at 1250s, wordy t32 rep3 at 285s). Bridge held the SSE stream open without emitting events; claw eventually killed for nominal timeout but the model never had a chance to finish.

**Implication:** Real book-store t32 = 3/3 = 100% (saturating). The 67% reading is a 5.5% bridge-transient rate masquerading as model failure. Documented in [bridge diagnostics memo](../../litellm/TODO-1.21-bridge-error-diagnostics.md) anomaly #2 for Sprint 1.5/2 to address.

**Fix (this sprint):** post-hoc filter — exclude `terminal_status=timeout AND join_status=stream_aborted_mid_run` rows from pass-rate calcs in `explore-summarize.mjs`. **Out-of-scope-for-1.21** — defer to bridge sprint.

### Q3 — Two-bucket t16 0/3 is real ctx-overflow, but iter-storm is the root cause

**Mechanism:** Snapshot `69b215bb-83cf-4c11-ad4e-580ffdcf757e` shows 14 iters captured (input_tokens trajectory 29-1665, no monotonic spike visible *in captured iters*). The screen log shows the 15th iter — never written to `iterations.jsonl` because claw exits on the 400 — hit `request (32934 tokens) > 32768 ctx`. The model edit-loops; each iter accumulates tool-result history; eventually one request tips over the 32k ceiling. The B-tweak's `JSON.stringify(r)` payload (~+200-500 tokens per failed verify) **inflated per-iter cost but did not cause the iter-storm**.

**Implication:** Reverting the B-tweak alone won't necessarily fix the t16 result. The model may still iter-storm into ctx at 32k, just slightly later. That outcome is FINE — it's an R9-A `ctx_discriminator` that counts toward the gate. If the revert lets the model converge in fewer iters and pass at t16, great — Tier B candidate. Either outcome moves the gate forward.

**Fix:** revert the `JSON.stringify(r)` payload; keep PROMPT worked example (lower-token cost, higher signal). See workstream "two-bucket revert" below.

## Lessons learned (improvements vs. before c3)

1. **`orphan_transcript_record` is universal**, not transient. We had been calling out individual cases of "empty trace" or "1-iter orphan" as failures; in fact, every run since cycle 1 has been silently orphan. This is a telemetry-only bug — runs complete fine, but per-iter timing breakdown (`server_decode_ms` etc.) is permanently null. Cataloged in the bridge memo as anomaly #1.
2. **t16 ctx-32k is a structural ceiling, not just an iter-budget reduction.** This is the basis of the Q2 user reframe (R9-A `ctx_discriminator` as a valid gate-counted class).
3. **B-tweaks need to address the right axis.** Bigger assertion messages help when the model can't read output (`result_changed_vs_previous_same_call: false`). They hurt when the model can read output but iter-storms (`true`). Distinguish before tweaking next time.
4. **`do not edit X` prompt instructions need enforcement.** Cascade-eight is the obvious case; review other tests (`semver-range`, others?) for similar prompt-only invariants the model might bypass.
5. **N=3 still has too much sample noise** to draw firm classifications. The c2 N=2 "noisy discriminators" framing was the right hedge, but we got over-confident on book-store (1 fail in N=2 c2 → assumed real failure → "clean discriminator"). c3 N=3 reveals book-store is consistent ctx-overflow, not noise. **Implication for Sprint 2:** N=60 confirmatory should hold the line; don't push it lower unless variance is empirically small.
