## Step 2 findings — empty-trace forensics (2026-05-03)

**Diagnosis: bridge/server transient, not model behavior.**

All three empty/near-empty c1.t32 traces (`grade-school`, `ini-parser`, `alphametics`-rep1) share the same upstream error in
[`logs/OVERNIGHT-SCREEN-explore-c1-20260502-1735.md`](../../logs/OVERNIGHT-SCREEN-explore-c1-20260502-1735.md):

```
[error-kind: unknown]
error: failed to parse Anthropic response for model anthropic/claw-llama: missing field `type` ...
{"error": {"message": "litellm.MidStreamFallbackError: litellm.APIConnectionError:
APIConnectionError: OpenAIException - Context size has been exceeded.",
"type": null, "param": null, "code": "500"}}
```

Two shapes branch from one root cause:
- `iter_count: 0` + `join_status: empty_run` (grade-school, ini-parser) — error fired on turn 1, before any iteration row was written. `iterations.jsonl` is literally 0 bytes; only the user prompt landed in the session file.
- `iter_count: 1` + `join_status: orphan_transcript_record` (alphametics rep 1) — error fired on turn 2, after the model's `read_file verify.js` came back. Iter 1 (5704 in / 28 out) was written before claw died.

The "Context size has been exceeded" message is **spurious**: measured turn-1 input was 5704 tokens for alphametics, and turn-2 would add only ~700 tokens of `verify.js` content (VERIFY_JS sizes: alphametics 2777 chars, book-store 2973, grade-school 4725, ini-parser 6272 — none anywhere near 65536 ctx). The error originates at the litellm/llama-server boundary as a 500, which claw cannot parse (no `type` field) and exits 1. The harness then correctly tags `terminal_status: error`, `passed: null`. Note `book-store` carries a heavier verifier than alphametics yet ran clean on the same chip — argues against "VERIFY_JS too large" as cause.

**Implication for tier classification.** The c1.t32 "errors" for grade-school and ini-parser are not model-fail signal; they are bridge transients that should be excluded. Re-running with the same model on the same prompt would almost certainly land another pass. Combined with c2.t16 = 100%/100% on both tests, the honest reading is **t32 saturates, t16 saturates** — these belong in Tier D (saturating), not Tier C (re-pilot).

**Verification gate revision.**

| Tier A confirmed (book-store, wordy) | 2 |
| Tier B if tweaks succeed (two-bucket, word-search) | +2 |
| ~~Tier C re-pilot~~ — collapses to Tier D | 0 |
| **Achievable without dipping** | **4/6** |
| **Gap** | 2 cells |

**Recommendation: SKIP cycle-3 re-pilot of grade-school + ini-parser.** Reclassify both as Tier D saturating. Two consequent actions in priority order:

1. **Off-chip B-tweaks (Step 4)** — proceed in full as drafted: tighten `two-bucket` and `word-search` verifier assertion messages + return-shape spec. These are the only realistic path to two more middle-band cells without authoring fresh tests.
2. **D-pool dip (was Step 5, now upgraded to Step 3)** — pick *one* of `cascade-eight 8→12 steps` or `twelve-file-refactor 12→18 files` to bump into the discriminative band, recovering one cell. If both bumps work cleanly, gate closes at 6/6 without re-piloting.
3. **Bridge transient open question (defer)** — log the spurious context-overflow 500 as a known intermittent. Rate observed: 3 cells out of ~36 attempted in c1 (≈8%). Tolerable for screen, but if it recurs at the N=60 confirmatory pilot we'll need to either harden claw's response parser or instrument the bridge. **Not blocking for c3.**

**Conclusion: "Harness fix X first" — but X is "no fix, no re-pilot; pivot to D-pool dip + B-tweaks."** Step 3 (chip dispatch on grade-school/ini-parser) is **cancelled**.

## Step 3+4 status (off-chip work, 2026-05-03)

User confirmed: skip c3 re-pilot of grade-school/ini-parser; proceed with D-pool dip + B-tweaks in parallel.

**D-pool dip (cascade-eight 8→9 files).** Already shipped in commit `e27a790` (cycle 1+2 commit) by analyze-agent: 9th file `i.js` adds an `indexOf`-vs-`includes` NaN trap, plus prior hardenings to `isEven` (zero-handling) and `sumArr` (string-concat masquerade). The fix-the-symptom heuristic now fails — model must recognize that `indexOf` uses `===` and switch to `Array.prototype.includes` (SameValueZero). **No additional off-chip work needed; bump is queued for next chip cycle to validate.**

**Tier B verifier-message tweaks.** Done this session.
- `two-bucket`: every assertion in `checkSolvable` now appends `'returned ' + JSON.stringify(r)` so the model sees its full output object on every failure (was: opaque `'(3,8,5,A): actionCount'`). Added a fully-worked example to PROMPT showing move-by-move state path for `(3,8,5,'A')`. Targets c2 `result_changed_vs_previous_same_call: false` iter-storm where the model rewrote 12 times without seeing what its output actually looked like.
- `word-search`: introduced a `check(label, expected, actual)` helper that on failure renders `expected ${expected} got ${actual}`. Consolidated the 5-word multi-target case (was 5 separate `deepEqual` calls debugged one-at-a-time) into a single combined assertion so the model sees all word locations + the missing-key sentinel together.
- Both verifiers validated against minimal post-mutation canonicals in `/tmp` (PASS) and against a deliberately-wrong solution to confirm the new error messages render the full result object.

**Verification gate, post-tweak.** Same arithmetic as before: best-case 4 (Tier A + Tier B if tweaks succeed) + 1–2 from cascade-eight bump if it crosses out of saturation = **5–6 cells.** Next chip cycle is the binding test.
